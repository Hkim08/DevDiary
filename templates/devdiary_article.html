How I Built an Automation Health Monitor System — and Packaged It as a Gumroad Product

Your n8n workflows run while you sleep. Your Make scenarios process invoices, sync CRMs, and fire emails — all without you. Until they don't. And you find out three hours later from an angry client. Here's how I built a monitoring layer to fix that, and then packaged everything as a Gumroad bundle.

01 — The ProblemThe Automation Tax Every Solo Dev Pays

I run about a dozen active automation workflows for client projects — invoice syncs, lead capture pipelines, report generators. They work beautifully. Until they don't.

The classic failure pattern: a webhook times out at 2am, three data records get dropped, and I find out the next afternoon when the client emails asking where their data is. By then, re-running the workflow creates duplicates. The fix takes an hour. The explanation takes longer.

The real cost of silent failures isn't just the bug — it's the compounding damage: lost data, broken trust, and the debugging tax on top of everything else you planned to ship that day.

What I needed wasn't a better workflow. I needed a monitoring layer that sits around my workflows and tells me immediately when something breaks — before the client notices.

02 — The ArchitectureA Closed Loop, Not a Dashboard

Most monitoring advice stops at "add logging." That's not enough. What you need is a closed loop: detect the failure, route an alert, log it, and have a playbook ready. The system I built has four stages:

The monitoring loop — from workflow failure to dashboard log in under 2 seconds.

  1. Monitor — error trigger hooks attached to every workflow, capturing execution context the moment something fails.
  2. Alert — structured Slack and Discord messages with severity levels, run IDs, retry counts, and playbook links.
  3. Log — every failure appended to a health dashboard (Google Sheets or Notion) for trend analysis.
  4. Fix — pre-written debugging playbooks for the four error types that cause 90% of automation failures.

The key insight: the alert is not the product. The alert is just the trigger. What saves time is having a playbook you can open immediately instead of spending 20 minutes re-diagnosing a problem you've already solved before.

03 — The CodeBuilding the Alert Scripts

Slack alert script

The Slack script uses Incoming Webhooks — no bot token, no OAuth dance, no app approval process. Just a URL and a POST request. I built it as a plain Node.js module with zero npm dependencies.

slack_alert_script.js 
// Call this from your workflow's error/catch node
await sendSlackAlert({
  workflowName:  'Invoice → CRM Sync',
  errorType:     'Webhook Timeout',
  errorMessage:  'Request timed out after 30s',
  severity:      'critical',
  runId:         'exec_1234',
  retryCount:    3,
  docsLink:      'https://yoursite.com/playbooks/webhook-timeouts',
});

The result in Slack is a structured message with a colored sidebar (red for critical, yellow for warning, green for info), a field grid showing workflow name, severity, run ID, and retry count, and a direct link to the relevant debugging playbook.

The alert payload — clean, structured, and directly actionable.

n8n integration

In n8n, the cleanest approach is a dedicated Error Workflow. Go to Settings → Error Workflow, create a new workflow, add a Function node, and call sendSlackAlert() with n8n's built-in error variables:

n8n Function node 
const { sendSlackAlert } = require('/path/to/slack_alert_script');

await sendSlackAlert({
  workflowName: $workflow.name,
  errorType:    $execution.error.name,
  errorMessage: $execution.error.message,
  severity:     'critical',
  runId:        $execution.id,
});
One error workflow, all workflows covered. Setting the Error Workflow at the n8n instance level means every workflow you create automatically gets monitoring — you don't have to remember to add error handling to each one.

Make and Zapier

For Make, add an HTTP module to the error route of each scenario. Set method to POST, URL to your Slack webhook, body type to Raw (JSON), and use Make's built-in error bundle variables: {"{"}error.message{"}"}, {"{"}error.type{"}"}.

For Zapier, use the built-in Error Handling under each Zap's Settings, pointing to a Webhooks action that posts to your alert endpoint.

04 — The DashboardTrack What You Can't See

Alerts are reactive. The dashboard is proactive. It gives you a weekly view of uptime percentages, error frequency trends, and which workflows are silently degrading before they fully break.

I built two versions: a Google Sheets dashboard (four tabs: Health Overview, Error Log, Alert Log, Weekly Summary) and a Notion equivalent with a board view grouped by workflow status.

What to track

  • Uptime % — calculated as (successful runs / total runs) × 100 over a rolling 7-day window
  • Error count (7d) — leading indicator; 3+ errors in a week means a workflow needs attention even if it's technically "healthy"
  • Alert sent flag — prevents duplicate alerts for the same recurring failure
  • Last run timestamp — catches silent stops where the workflow simply ceases to execute
The metric that matters most is not uptime % — it's error frequency. A workflow at 99% uptime with 8 errors in a week is a workflow about to break completely. Uptime can look healthy right up until it doesn't.

05 — The PlaybooksDebug Once, Never Again

The debugging playbooks are the real force multiplier in this system. Every time I encounter a new error type and solve it, I write it down in structured format: symptoms → root causes → step-by-step fix → prevention. Four playbooks cover 90% of what I see in the wild:

  • Webhook Timeouts — curl test, retry with exponential backoff, timeout threshold configuration
  • API Rate Limits — parse Retry-After header, implement token-bucket throttle, batch API calls
  • JSON Parsing Errors — safe parse with fallback, schema validation, optional chaining for nulls
  • Auth Failures — token refresh logic, credential rotation schedule, least-privilege scopes

Each playbook has a root cause probability table so you start with the most likely fix, not the most interesting one. This alone cuts debugging time by about 60% — you're not re-diagnosing, you're executing a known solution.

07 — LessonsWhat I'd Do Differently

Build for the failure you haven't seen yet

The most useful playbook I wrote was the one I created before I encountered the error — by reading other developers' post-mortems and formalizing what I'd do if it happened to me. Proactive documentation beats reactive firefighting.

The dashboard is a commitment device

Having a dashboard you update weekly forces you to actually look at your automation health. Without it, "everything is fine" is just an assumption. With it, you catch the workflow that's been at 91% uptime for two weeks and needs attention before it hits 0%.

SQLite over Google Sheets for production

The Google Sheets dashboard is great for visual review. But if you're logging hundreds of events per day, you want a local SQLite database as the source of truth and only push summary stats to Sheets. Keep the operational data close to where it's generated.
